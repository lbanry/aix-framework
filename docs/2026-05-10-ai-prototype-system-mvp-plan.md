# AI Prototype System MVP Implementation Plan

Date: 2026-05-10

## Goal

Build the first vertical slice of a design-system-configurable prototype generation system inside the existing AIX framework repo.

The MVP should prove this path:

```text
interface requirement + interface system + research findings
  -> structured interface plan
  -> deterministic static HTML prototype
  -> machine-readable validation report
  -> local preview
```

This is intentionally narrower than the full design-system-agnostic architecture in the working log. The MVP should reuse the existing `@aix/design` package and extend the current `aix interface` workflow rather than creating a second product surface.

## Repo Policy

Prototype source generated by the tool remains local-only by default.

- Keep framework implementation, schemas, fixtures, examples, and docs in Git.
- Keep generated prototype websites under `prototypes/`, which is already gitignored.
- Do not commit generated prototype output unless explicitly approved.
- Do not commit `image-references/`, OCR output, screenshots, or local preview artifacts unless explicitly approved.

## MVP Scope

### In Scope

- One deterministic static HTML generator.
- One normalized prototype manifest schema.
- One pattern registry backed by the existing interface system and interface plan shape.
- One local design-system adapter path using existing AIX interface system YAML.
- One generated output format: static `index.html` plus optional `prototype.json`.
- One validation command that checks generated HTML against the source plan and system.
- One local preview command.
- Smoke tests for generation and validation.

### Out Of Scope

- Figma API ingestion.
- Style Dictionary ingestion.
- Tokens Studio ingestion.
- LLM-driven component invention.
- Auto-promotion of feedback logs into patterns.
- Git hooks.
- Multi-page app routing.
- Full dependency stack from the source document.
- Committing generated prototype output.

## Package Boundary

Add prototype generation to `packages/aix-design` because it is design-system-aware and interface-specific.

Expected source layout:

```text
packages/aix-design/
  spec/
    prototype-manifest.schema.json
    prototype-validation-report.schema.json
  src/
    prototype/
      manifest.js
      generator.js
      render-html.js
      validate.js
      preview.js
      file-output.js
  interface/
    prototype-patterns/
      static-html.yaml
```

Keep `packages/aix-core` unchanged for this MVP unless a shared utility is clearly required.

## CLI Surface

Extend the existing `aix interface` command group:

```bash
aix interface prototype scaffold <plan> --system <system> --out prototypes/<name>
aix interface prototype verify <prototypeDir> --plan <plan> --system <system> --json
aix interface prototype dev <prototypeDir> --port 4173
```

Initial behavior:

- `scaffold` reads an existing interface plan and interface system, creates a normalized prototype manifest, renders static HTML, and writes into the requested local output directory.
- `verify` reads generated HTML plus the manifest and emits a validation report.
- `dev` serves the generated directory locally for review.

The command names intentionally sit under `aix interface` because this is an extension of the existing interface planning pipeline.

## Data Contracts

### Prototype Manifest

Create `packages/aix-design/spec/prototype-manifest.schema.json`.

Minimum fields:

```yaml
version: 0.1.0
screen_id: contract-inspection-review
source_plan: packages/aix-design/interface/plans/contract-inspection.plan.yaml
source_system: packages/aix-design/interface/systems/aix-interface-system.yaml
output:
  format: static_html
  entrypoint: index.html
tokens:
  color:
    - status.pass
  spacing:
    - regular
patterns:
  - id: inspection_results
sections:
  - id: readiness-score
    title: Readiness Score
    pattern: inspection_results
    components:
      - ReadinessScore
    requirements:
      - readiness_score
validation_rules:
  - prototype.plan.section-mapped
  - prototype.system.component-approved
  - prototype.html.semantic-landmarks
  - prototype.html.accessible-buttons
  - prototype.tokens.no-unknown-token
```

### Validation Report

Create `packages/aix-design/spec/prototype-validation-report.schema.json`.

Minimum fields:

```yaml
valid: true
prototype_dir: prototypes/contract-inspection-review
findings:
  - rule_id: prototype.plan.section-mapped
    severity: error
    message: Section is missing from generated HTML.
    selector: '[data-aix-section="readiness-score"]'
    source:
      plan_section: readiness-score
summary:
  errors: 0
  warnings: 0
```

Rule IDs must be stable from the beginning so reports can become automation-friendly later.

## Generator Behavior

The MVP generator should be deterministic and conservative.

1. Read and validate the interface plan with the existing plan schema.
2. Read and validate the interface system with the existing system schema.
3. Convert the plan into a prototype manifest.
4. Render a static HTML document with:
   - semantic landmarks: `header`, `main`, `section`, `footer` where useful
   - `data-aix-screen`, `data-aix-section`, `data-aix-pattern`, and `data-aix-component` attributes
   - visible text for all plan sections
   - basic keyboard-safe controls only when the plan calls for an action
   - CSS custom properties generated from approved token names
5. Write:
   - `index.html`
   - `prototype.json`
   - `validation-report.json`

For placeholder content, use plan-derived labels and requirement IDs. Do not invent product-specific copy beyond what exists in the plan.

## Validation Rules

Implement the first validators without adding heavy dependencies.

Required MVP rules:

- `prototype.plan.screen-mapped`: generated HTML has `data-aix-screen` matching `plan.screen_id`.
- `prototype.plan.section-mapped`: every plan section has one matching `data-aix-section`.
- `prototype.system.pattern-approved`: every rendered pattern exists in the source system.
- `prototype.system.component-approved`: every rendered component exists in the source system.
- `prototype.html.semantic-main`: generated HTML has exactly one `main`.
- `prototype.html.heading-order`: generated headings do not skip levels.
- `prototype.html.button-name`: every `button` has accessible text.
- `prototype.tokens.no-unknown-token`: generated CSS token references are declared by the source system.

Defer `axe-core`, `jsdom`, `cheerio`, `postcss`, and `htmlhint` until these rules become too brittle with simple parsing. The first pass can use deterministic renderer metadata plus lightweight HTML checks.

## Initial MVP Milestone Definition

The MVP is complete when all of the following are true:

- `aix interface plan` still works as it does today.
- `aix interface prototype scaffold` generates a static local prototype from `contract-inspection.plan.yaml`.
- `aix interface prototype verify --json` returns a valid report with stable rule IDs.
- `aix interface prototype dev` serves the generated prototype locally.
- `npm test` covers the new command path.
- No generated `prototypes/` files are tracked by Git.
- The implementation does not require Figma, OCR, browser automation, or an LLM call.

## First Implementation Order

1. Add prototype schemas and manifest generation.
2. Add static HTML rendering and file output.
3. Add CLI scaffold command.
4. Add validation rules and JSON report output.
5. Add smoke tests.
6. Add preview server.
7. Update docs and run repo policy checks.

This order keeps the core product contract visible early and avoids building a preview server before the generated artifact and validator are stable.
